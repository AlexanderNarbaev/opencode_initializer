# FAQ — Часто задаваемые вопросы

## Установка и первый запуск

### Сколько времени занимает установка?

Полная установка (`--full`) на свежей Ubuntu 22.04/24.04 занимает **12–18 минут** на SSD и быстром интернете. На HDD или медленном соединении — до 30 минут. Интерактивный режим (`--interactive`) — от 3 до 10 минут, в зависимости от выбранных компонентов. Скрипт показывает прогресс-бар (19 шагов), текущий этап и оставшееся время.

### Что делать, если установка прервалась?

Перезапустите ту же команду — **прогресс сохраняется** в `~/.cache/opencode-setup/progress`. Скрипт определит, какие шаги уже выполнены, и продолжит с места остановки. Если прогресс-файл повреждён, запустите `bash setup.sh --reinit` — он переустановит инструменты, сохранив конфигурацию и секреты.

### Как установить или переустановить только один компонент?

Используйте CLI `dev`: `dev install docker`, `dev install llm`, `dev install java`. Для удаления: `dev remove java`. Если нужно переустановить конкретный компонент — удалите его через `dev remove`, затем `dev install`. Список доступных компонентов: `dev list`.

### Нужно ли клонировать репозиторий?

Нет. Команда `bash <(curl -fsSL ...) --full` автоматически клонирует репозиторий в `~/.cache/opencode-setup/repo`, устанавливает CLI `dev` и запускает сборку. После первой установки можно использовать локальный скрипт: `~/setup.sh --health` или `dev health`.

## WSL2

### Почему не работает Docker в WSL2?

Docker Desktop должен быть установлен **в Windows**, а не внутри WSL2. Скрипт проверяет наличие `docker.exe` в Windows PATH и настраивает WSL2-интеграцию автоматически. Если Docker не найден — запустите `dev install docker` из WSL2, он доустановит Docker Engine внутри дистрибутива. Проверьте, что в настройках Docker Desktop включена интеграция с вашим дистрибутивом (Settings → Resources → WSL Integration).

### Почему не работает GPU (Ollama/vLLM) в WSL2?

Требуется WSL2 с ядром 5.15+ и драйвер NVIDIA для WSL2 (версия 525+). Установите драйвер с [nvidia.com](https://www.nvidia.com/download/). Проверьте: `nvidia-smi` — если выводит информацию о GPU, всё настроено. Если нет — обновите WSL2: `wsl --update` из PowerShell, затем перезапустите WSL.

### Как настроить сеть в WSL2 для России?

Скрипт автоматически добавляет DNS 8.8.8.8, 1.1.1.1 и 77.88.8.8 (Яндекс) в `/etc/resolv.conf`, а также настраивает зеркала для GitHub, npm, pip, Docker, Go. Если сайты недоступны — проверьте `/etc/resolv.conf`: `cat /etc/resolv.conf`. При необходимости запустите `bash setup.sh --health` — он в секции DNS покажет текущее состояние.

## API ключи

### Где взять ключи для AI-провайдеров?

| Провайдер | Где получить ключ |
|-----------|------------------|
| DeepSeek | [platform.deepseek.com](https://platform.deepseek.com/) — регистрация, пополнение баланса |
| OpenCode Go | [opencode.ai/auth](https://opencode.ai/auth) — встроенный провайдер |
| xAI Grok | [console.x.ai](https://console.x.ai/) — требует верификации телефона |
| Moonshot | [platform.moonshot.cn](https://platform.moonshot.cn/) |
| MiniMax | [platform.minimax.io](https://platform.minimax.io/) |

Ключи передаются через аргументы командной строки: `--deepseek-key "sk-..."`, `--xai-key "xai-..."`, `-k "opencode-api-key"`. Сохраните команду в защищённом месте.

### Что если нет ключа DeepSeek? Можно ли использовать только OpenCode Go?

Да. Если передан только ключ OpenCode (`-k`), скрипт настроит его как единственного провайдера. В `opencode.json` будет только секция `"opencode"`. При отсутствии всех ключей — скрипт предупредит, что модели не будут доступны, но установка инструментов продолжится.

### Безопасно ли хранить ключи в secrets.env?

Да, при условии: файл `~/secrets.env` имеет права `chmod 600` (только владелец), не коммитится в git (добавлен в `.gitignore` скрипта), НЕ сорсится в `.zshrc` или `.bashrc`. Ключи передаются в `opencode.json` как значения переменных окружения из `secrets.env`. Никогда не добавляйте `secrets.env` в shell-rc файлы — это основная причина утечек.

## Обновление

### Как обновить инструменты и сам скрипт?

- **`dev update`** — обновляет npm-пакеты (opencode-ai, MCP-серверы, плагины), uv, системные пакеты, запускает миграции и перегенерирует `opencode.json`.
- **`dev self-update`** — `git pull` + обновление symlink `~/setup.sh` + переустановка CLI `dev`. Обновляет только код скрипта, не трогает инструменты.
- **`dev autoupdate`** — запускает `topgrade` (полное обновление ОС, пакетов, pip, npm, cargo, и т.д.). По умолчанию настроен systemd-таймер на воскресенье 04:00.

Рекомендуемый порядок: `dev self-update`, затем `dev update`.

### Почему после обновления что-то сломалось?

Самая частая причина — изменения в формате `opencode.json`. Запустите `bash setup.sh --fix-config` — он перегенерирует конфиг с текущими путями и настройками. Если не помогло — `dev health` покажет, что именно сломалось. Для полной переустановки конфигурации (сохраняя данные): `bash setup.sh --reinit`.

### Что делают миграции?

Миграции в `migrations/` — это timestamped-скрипты, автоматически запускаемые при `dev update`. Каждая выполняется однократно (флаг в `~/.cache/opencode-setup/.mig-YYYYMMDD`). Миграции исправляют проблемы совместимости между версиями: обновление схемы `opencode.json`, удаление несовместимых плагинов, исправление путей. Просмотреть выполненные: `ls ~/.cache/opencode-setup/.mig-*`.

## OpenCode

### Как сменить модель (например, на более дешёвую)?

Отредактируйте поле `"model"` в `opencode.json` (находится в `/home/$USER/opencode.json` или через `dev config`). Примеры:
- DeepSeek V4 Flash (дешевле, быстрее): `"deepseek/deepseek-v4-flash"`
- OpenCode Go: `"opencode/opencode-go"`
- xAI Grok: `"xai/grok-3"`
- Локальная Ollama: `"ollama/qwen3:30b"`

Для маленькой модели (автодополнения, быстрые ответы) — поле `"small_model"`. После изменения перезапускать OpenCode не нужно, изменения применяются автоматически.

### Как добавить свой MCP-сервер?

Добавьте запись в `opencode.json` → `"mcp_servers"`. Пример для кастомного сервера:
```json
"my-server": {
  "type": "local",
  "command": ["node", "/path/to/my-server/dist/index.js"],
  "env": { "API_KEY": "${MY_API_KEY}" }
}
```
Для remote-сервера (HTTP/SSE):
```json
"my-remote": {
  "type": "remote",
  "url": "http://localhost:3001/sse"
}
```
Переменные окружения берите из `~/secrets.env`. Запустите `dev health` чтобы проверить, что сервер стартует.

### Как отключить ненужный плагин?

В `opencode.json` найдите секцию `"plugins"`. Удалите или закомментируйте строку с названием плагина. Например, чтобы отключить `opencode-vibeguard`:
```json
"plugins": [
  "opencode-codegraph",
  // "opencode-vibeguard",
  "opencode-orchestrator"
]
```
**Важно**: используйте `//` для комментариев — `opencode.json` поддерживает JSONC. После изменений перезапустите OpenCode.

### OpenCode не видит мои ключи / модель не найдена

Проверьте `cat ~/secrets.env` — ключи должны быть там. Проверьте `opencode.json` — в секции `"provider"` должны быть соответствующие провайдеры. Запустите `bash setup.sh --fix-config` для перегенерации конфига с текущими ключами. Проверьте формат ключа DeepSeek — он должен начинаться с `sk-`.

## ZSH

### Почему не работает автодополнение?

Проверьте версию ZSH: `zsh --version` (должна быть ≥ 5.8). Если версия старая — скрипт должен был обновить её автоматически, но на некоторых старых дистрибутивах это невозможно. Проверьте, что плагины загружены: `echo $plugins` в сессии zsh. Запустите `exec zsh` чтобы перезагрузить оболочку. Если не помогло — `bash setup.sh --fix-zshrc`.

### Как вернуть старый .zshrc?

Скрипт создаёт бекап перед модификацией: `cp ~/.zshrc ~/.zshrc.bak.$(date +%Y%m%d%H%M%S)`. Список бекапов: `ls -la ~/.zshrc.bak.*`. Для восстановления: `cp ~/.zshrc.bak.20250101120000 ~/.zshrc && exec zsh`. Если бекап не найден — `bash setup.sh --fix-zshrc` перегенерирует рабочую конфигурацию.

### Плагины ZSH выдают ошибки при старте

Проверьте, что Oh My Zsh установлен: `ls ~/.oh-my-zsh/`. Проверьте custom-плагины: `ls ~/.oh-my-zsh/custom/plugins/`. Если какой-то плагин не найден — `bash setup.sh --fix-zshrc` переустановит стандартный набор. Ошибки `compinit` обычно решаются командой `rm -f ~/.zcompdump* && exec zsh`.

## Производительность

### Почему OpenCode медленно отвечает?

Причины по убыванию вероятности: (1) Медленный API-провайдер — проверьте `ping api.deepseek.com`, попробуйте переключиться на OpenCode Go. (2) Много холодных MCP-серверов — скрипт использует `bun bin paths` вместо `npx -y`, что ускоряет холодный старт с 3–5 сек до <1 сек. Проверьте: `which c7-mcp-server`. (3) Большой проект без индексации — запустите codegraph: `opencode codegraph serve --mcp`.

### Как ускорить MCP-серверы?

Убедитесь, что MCP-серверы запускаются через Bun, а не Node: `which c7-mcp-server` должен показывать путь в `~/.bun/bin/`. Если нет — `bash setup.sh --reinit` переустановит их через Bun. Для проверки времени холодного старта: `time c7-mcp-server --version`. Норма — менее 1 секунды.

### Куда девается место на диске?

Кеш хранится в `~/.cache/opencode-setup/`: npm-пакеты (`.tgz`), репозиторий, прогресс. Очистка: `rm -rf ~/.cache/opencode-setup/`. Логи: `~/setup-*.log` — можно удалить старые. Docker: `docker system prune -af` — очищает неиспользуемые образы и контейнеры. Ollama: модели хранятся в `~/.ollama/models/`, удаление: `ollama rm <model>`.

## GPU / LLM

### Нужен ли GPU для работы?

**Нет.** OpenCode работает с облачными API (DeepSeek, OpenCode Go, xAI) — GPU не требуется. Локальные LLM (Ollama, vLLM, SGLang) — опциональны и требуют GPU. На ноутбуке без GPU можно запускать маленькие модели (1–3B параметров) в Ollama на CPU, но скорость будет низкой.

### Какие модели запускаются локально через Ollama?

Скрипт устанавливает Ollama как systemd-сервис. Рекомендуемые модели:
- **GPU 8+ GB VRAM**: `qwen3:30b` (30B), `llama3.1:8b` (8B), `mistral:7b` (7B)
- **GPU 4–6 GB**: `qwen3:4b` (4B), `phi4:3.8b` (3.8B)
- **CPU only**: `qwen3:1.8b` (1.8B), `phi4-mini:1.7b` (1.7B)

Установка модели: `ollama pull qwen3:8b`. Смена модели в OpenCode: `"model": "ollama/qwen3:8b"` в `opencode.json`.

### Ollama vs vLLM — что выбрать?

**Ollama** — простая установка (один бинарник), авто-определение GPU, удобный CLI. Подходит для разработки и экспериментов. Ограничение: одна модель одновременно.

**vLLM** — production-grade сервер, поддержка множества моделей параллельно, PagedAttention для эффективного использования VRAM, OpenAI-совместимый API. Требует больше настройки, рекомендуется для серьёзных проектов.

Для начала работы хватит Ollama. Если нужно запускать 2+ модели одновременно или production-окружение — vLLM.

### Почему модель не скачивается через Ollama?

Проверьте интернет: `curl -I https://ollama.com`. Проверьте свободное место: `df -h ~/.ollama/models/`. Для больших моделей (30B+) требуется 20+ GB свободного места. Если модель зависает на загрузке — попробуйте другую модель меньшего размера: `ollama pull phi4:3.8b`. Для России релевантно: используйте VPN или прокси, если прямой доступ к ollama.com заблокирован.
