## 2. RAG-инфраструктура

### Сравнение векторных БД

| БД | RAM | Скорость (10M векторов) | Сложность | Офлайн | Для чего |
|----|-----|------------------------|-----------|--------|----------|
| **ChromaDB** | 200MB | 50ms | Минимальная | ✅ ONNX | Память агента |
| **Qdrant** | 2GB+ | 10ms | Средняя | ⚠️ Модели отдельно | Корпоративный RAG |
| **Weaviate** | 4GB+ | 15ms | Высокая | ⚠️ Модели отдельно | Enterprise |
| **Milvus** | 8GB+ | 5ms | Очень высокая | ❌ | Big Data |
| **FAISS** | N/A (in-memory) | 1ms | Низкая (но нет персистентности) | ✅ | Эксперименты |

### Почему ChromaDB для локальной разработки

- **ONNX-модель** (all-MiniLM-L6-v2, 79MB) работает **полностью офлайн** — не требует интернета и API-ключей
- **Systemd-сервис** — запускается автоматически при логине, не требует ручного управления
- **200MB RAM** — минимальное потребление, подходит для ноутбуков
- **Muninn-интеграция** — ChromaDB нативный бэкенд для Muninn

### Почему Qdrant для корпоративного использования

- **RBAC** — изоляция пространств имён по отделам
- **Горизонтальное масштабирование** — шардирование, 100M+ векторов
- **Фильтрация по метаданным** — дата, автор, проект
- **Rust** — производительность на уровне C++

### Гибридный поиск: dense + sparse = +35% точности

Комбинация Qdrant (плотные векторы) + BM25 (разреженные векторы) даёт прирост точности на 35% по сравнению с чисто семантическим поиском. RAGFlow реализует обе стратегии из коробки.

## 3. Безопасность AI-агентов

### Векторы атак (классификация)

| Вектор | Вероятность | Последствия | Защита |
|--------|------------|-------------|--------|
| Утечка секретов через промпт | Высокая | Компрометация API-ключей | stranger-danger |
| Prompt injection | Средняя | Выполнение произвольных команд | stranger-danger |
| Деструктивные команды | Низкая | Потеря данных/системы | damage-control (144 паттерна) |
| Чтение чувствительных файлов | Средняя | Утечка .env, .ssh | damage-control (блокировка путей) |
| Эксплойты через сгенерированный код | Средняя | SQL-инъекции, XSS | Trivy + Qodana |
| DoS через переполнение контекста | Высокая | Рост затрат, падение качества | dcp (Dynamic Context Pruning) |

### Многоуровневая защита

```
Запрос пользователя
    ↓
[Уровень 1] stranger-danger: фильтрация PII, prompt injection
    ↓
[Уровень 2] damage-control: блокировка опасных команд/путей
    ↓
[Уровень 3] codegraph: структурное понимание кода (безопасный анализ)
    ↓
[Уровень 4] dcp: сжатие контекста (защита от DoS)
    ↓
[Уровень 5] auto-fallback: переключение провайдера при сбое
    ↓
Выполнение
```

### Статистика галлюцинаций

| Тип задачи | DeepSeek V4 Pro | Qwen3-Coder-14B | Claude 4 | GPT-5 |
|-----------|----------------|-----------------|----------|-------|
| Генерация кода | 3.4% | 5.2% | 2.8% | 2.1% |
| API/библиотеки | 11.8% | 18.3% | 13.5% | 15.2% |
| Бизнес-логика | 5.2% | 9.8% | 4.1% | 4.7% |

**Вывод:** Context7 MCP снижает галлюцинации API/библиотек с 15-18% до ~5%.

### Методы снижения галлюцинаций (по эффективности)

1. **RAG с официальной документацией** — снижение на 60-70% (Context7 + RAGFlow)
2. **Self-consistency (голосование 3 генераций)** — 50-60%, но ×3 стоимости
3. **Chain-of-thought** — 30-40% (sequential-thinking MCP)
4. **Few-shot примеры** — 20-30%
5. **Снижение температуры** — 10-20%, но теряется креативность

## 5. Плагины — пороги и настройки

### opencode-dcp: расчёт порогов сжатия

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| minContextLimit | 20,000 | Ниже — свежий диалог, сжатие не нужно |
| maxContextLimit | 48,000 | Выше — агрессивное сжатие старых сообщений |
| protectTokens | 40,000 | Сохранять последние 40K токенов |
| minTokens | 20,000 | Минимум для сохранения смысла при сжатии |
| autoCompaction | true | Автоматически при достижении порога |

**Почему 48K:** компромисс между качеством и стоимостью. 48K токенов × $2/1M вход = $0.096 за запрос к DeepSeek V4 Pro. 128K контекст стоил бы $0.256 — в 2.7 раза дороже.

## 6. SGLang vs vLLM для инференса

| Критерий | SGLang | vLLM |
|----------|--------|------|
| Структурированная генерация (JSON) | ✅ Нативная | ⚠️ Ограниченная |
| RadixAttention (кэширование префиксов) | ✅ | ❌ |
| Multi-node | ⚠️ Ручная настройка | ✅ Ray-based |
| Chunked prefill (длинный контекст) | ✅ | ✅ |
| Лучше для | Single-GPU, structured output | Кластер, throughput |

### Проблема длинного контекста на 8GB GPU

Принудительный 100K контекст на RTX 3070 8GB приводит к spill KV-кэша в системную RAM. Пропускная способность PCIe 4.0 ×16 = 32GB/s против GPU HBM = 1TB/s — падение скорости в 30 раз. Решение: RAG + нативный 32K контекст вместо принудительного 100K.

## 7. CI/CD для AI-генерируемого кода

### Трёхэтапное ревью

```
Commit → [1] Автоматическое (linter, typecheck, tests) → 5 мин
       → [2] AI-ревью (n8n + Qwen3-Coder) → 15 мин
       → [3] Человеческое ревью → по расписанию
```

**Статистика AI-ревью на Java/Spring Boot проектах:**
- 80% стилевых замечаний отлавливается до human review
- 45% логических ошибок детектируется моделью 14B
- 15% false positive (ревьюер учится игнорировать)

### n8n workflow для AI-ревью

```
GitLab Webhook (MR created)
  → GitLab API (получить diff)
  → RAGFlow (получить конвенции проекта)
  → Qwen3-Coder-14B (анализ diff)
  → GitLab API (опубликовать ревью)
```

## 9. Архитектура памяти агента

```
┌──────────────────────────────────────────────┐
│              СЕССИЯ (working memory)          │
│  MemoryLayer MCP: remember, recall, reflect  │
│  Agentic Tools MCP: задачи, прогресс         │
└──────────────────┬───────────────────────────┘
                   │ commit
┌──────────────────▼───────────────────────────┐
│           ДОЛГОВРЕМЕННАЯ ПАМЯТЬ              │
│  ChromaDB: векторное хранение                │
│  Muninn: семантическая память сессий         │
│  WAL: состояние модулей проекта              │
└──────────────────────────────────────────────┘
```

### Разделение зон ответственности

| Компонент | Что хранит | Формат | Когда использовать |
|-----------|-----------|--------|-------------------|
| MemoryLayer | Факты, решения, диалоги | Эпизодическая + семантическая | Поиск по смыслу |
| Agentic Tools | Задачи, подзадачи, прогресс | Иерархическая | Планирование |
| ChromaDB | Векторные эмбеддинги | 384-мерные векторы | Семантический поиск |
| Muninn | Сессионный контекст | Сырые тексты + метаданные | Продолжение сессии |
| WAL | Состояние проекта | Markdown | Человекочитаемый лог |
